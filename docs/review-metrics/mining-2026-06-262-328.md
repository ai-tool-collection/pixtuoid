## Architecture-drift (system lens) — PASS (new leg)

Architecture held under heavy in-window growth (9 sources incl. a daemon lane,
a doctor health board, source-gating) with no structural crack. Verified
against **HEAD**, not just docs.

**Module decomposition — still sound.** Crate split still strict
producer→reducer→renderer; dep direction one-way (pixtuoid→pixtuoid-core;
pixtuoid-hook depends on neither — confirmed in all three Cargo.toml). Two new
module-level seams earned their keep:

1. **source/daemon.rs** as a shared daemon-agnostic presence layer
   (`DaemonPresenceUpdate` vocab + `apply_presence`/sweep + `PresenceExitWatch`),
   with per-daemon WIRE decode staying in openclaw.rs — the daemon's state
   machine is generic, only its wire format is source-specific (mirrors the
   agent Source seam keeping decoders per-source).
2. **install/target.rs** `Target` is now a fully-typed exhaustive struct
   (`core_source`, `verify_schema` fn, `BinaryStrategy` enum, `extra_artifacts`
   Option). The window collapsed two bool-soups
   (`needs_path_warning`/`needs_resolved_binary` → `BinaryStrategy`,
   R0615-CR5; dead `restart_noun`/`post_install_note` removed, R0615-CR4) and
   made `verify_schema` a **required** field — so a new source physically
   cannot ship without wiring the doctor health board (the OCB4/OC4
   won't-compile merge-blocker is this enforcement working). `Target` is the
   single bridge registry/source-gating/doctor join on via `core_source` (not
   name — the #283 `claude-code`/`claude` join bug is permanently fixed).

One concern, not a defect: source/registry.rs's doc-block (and the nested
core/CLAUDE.md Layout entry) is a 30+-line wall describing
daemon.rs/hook/router.rs/lock-arbitration inline — accurate but at the edge of
readable, a candidate for the next doc-slim (cf. #270's 251→197 root slim).

1. **Shared `_pid` field, 3 producers / 2 consumers at one socket.** CodeWhale
